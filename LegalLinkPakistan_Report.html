LegalLink Pakistan

Complete Architecture & Core Process Flow Report
A detailed structural documentation of the LegalLink platform, including user pathways, real-time messaging, payments, and verification cycles.
DEVELOPER: Antigravity AI
DATE: July 2026
VERSION: 1.1.0

1. Executive Summary

LegalLink Pakistan is a premium, real-time mobile legal consultation platform connecting clients with verified legal practitioners in Pakistan. The application supports end-to-end appointment scheduling, automated payment routing via Stripe Connect, live video/voice consultations, and a fully functional WhatsApp-style messaging experience including audio notes, file transfers, and message deletions.

2. User Registration & Activation

The system segregates access levels into three distinct roles: Client, Lawyer, and Administrator. Each path operates its own unique sequence:

CLIENT

Client Onboarding

1
Form Submission
Client enters general parameters: Name, Email, Phone, Province, District, Tehsil, and Password.
2
API Authentication
Submits a JSON payload to the /api/auth/register-client route. The user account is saved to MongoDB.
3
Success Redirect
Direct login access. No admin approval required. Ready to browse lawyers.
LAWYER

Lawyer Onboarding & Verification

1
Basic Information (Step 1/3)
Lawyer supplies Name, Email, Password, Province, District, and Court Level.
2
Professional Verification (Step 2/3)
Inputs Bar Council Name, Enrollment Number, License Number, and uploads a clear picture of the License copy.
3
Areas of Expertise & Base64 Upload (Step 3/3)
Selects practice areas, writes a bio, inputs consultation fee, and uploads a profile picture. The app sends the request as a JSON payload using base64 file encodings.
4
Admin Review Hold
The lawyer record is written to MongoDB with status: 'pending' and isApproved: false. The lawyer is redirected to a static pending screen and cannot accept bookings.

3. Admin Verification Cycle

Pending lawyers are verified by the Admin dashboard in a secure, authenticated pathway:

ADMINISTRATOR

Verification Pipeline

1
Pending List Load
Admin loads the requests list. The app requests /api/admin/pending-lawyers with the stored adminToken.
2
Detailed Check
Admin clicks on a lawyer. The app fetches their specific profile and high-res license copy.
3
Approval/Rejection Update
Admin clicks Approve or Reject. The app sends a POST to /api/admin/update-status with the lawyer ID, final status, and any optional rejection reasons.

4. Booking & Consultation Flow

Once a lawyer is approved, clients can initiate paid bookings:

Step Action System Handling
1. Search & Select Client selects an approved lawyer. App fetches approved lawyers.
2. Scheduling Client picks Date and Time Slot. Creates a booking with status 'pending'.
3. Payment Intent Stripe card checkout form loads. Requests Stripe payment intent secret from backend.
4. Confirmation Payment completes successfully. Backend updates booking status to 'confirmed' and paymentStatus to 'paid'. Triggers push notifications.
5. Direct Interaction Consultation slot becomes active. Enables the real-time chat screen, voice note capabilities, and video/voice calling access.

5. High-Fidelity Communication Suite

The messaging screen replicates premium WhatsApp-style interactions:

Communication Systems Architecture

  • Dynamic Messaging Layout: Light blue bubbled structure (#e1eafd for sender) with read receipts and timestamps.
  • Voice Note System: Records sound files natively, updates a blinking microphone layout pill with timer increments, and saves the file on the backend uploads path.
  • Multi-Message Deletion: Long-pressing messages launches a WhatsApp-style selection bar showing the selected message count.
    • If the selected messages are all sent by the current user, they are given options: "Delete for me" and "Delete for everyone".
    • If any selected message was sent by the other party, only "Delete for me" is available.
  • WebRTC Voice & Video Calling: Includes circular caller avatars, animated pulse waves, and structured controls (hangup, speaker, microphone mute).

6. Database Schema Summary

The database runs on MongoDB and maps out relational endpoints using Mongoose models:

Model Key Fields Relationships
Client name, email, phone, address, profilePicUri, password Has many Bookings.
Lawyer name, email, phone, status, isApproved, licensePicUri, profilePicUri, areasOfPractice Has many Bookings, belongs to District/Province.
Booking clientId, lawyerId, date, timeSlot, status, paymentStatus, review, deletedBy References Client and Lawyer.
Message bookingId, sender, text, fileUrl, duration, type, deletedBy References Booking.
